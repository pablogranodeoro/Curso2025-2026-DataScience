Análisis de la Licencia del Conjunto de Datos

1. Publicador y titular de derechos

El conjunto de datos original ha sido publicado por una organización del Ayuntamiento de Alicante, por lo que se puede considerar que su titular de derechos es el propio organismo público responsable de su publicación.

2. Licencia del conjunto de datos original

La licencia del dataset original permite:

  • Compartir — copiar y redistribuir el material en cualquier medio o formato, incluso con fines comerciales.
  • Adaptar — remezclar, transformar y construir a partir del material para cualquier propósito, incluso comercial.

Bajo las siguientes condiciones:

  • Atribución — se debe dar crédito adecuado, proporcionar un enlace a la licencia e indicar si se realizaron cambios.
  • No aplicar restricciones adicionales que impidan legalmente a otros utilizar el material según la licencia.

Explicación: La licencia permite un uso amplio de los datos originales, incluso con fines comerciales o educativos, siempre que se cite correctamente la fuente y no se impongan restricciones adicionales.

3. Licencia propuesta para el conjunto de datos generado

Los datos procesados y generados por nuestro grupo se publicarán bajo la licencia Creative Commons Attribution 4.0 International (CC BY 4.0), lo que permitirá que otros los utilicen, compartan y modifiquen, siempre reconociendo la autoría del grupo.

Explicación: Dado que nuestro dataset contiene valores incompletos o inconsistentes, es importante permitir modificaciones. Esta licencia asegura que los datos puedan ser reutilizados y corregidos por terceros, fomentando su aprovechamiento ético y responsable.

3. Estrategia de Nombrado de Recursos (URI)

La estrategia de nombramiento se define siguiendo las directrices de la clase (Separación de Modelo e Instancias, Persistencia, Unicidad) y se basa exclusivamente en el modelo ontológico (prefijo ns) definido en el archivo ontology.ttl. La autoridad para las instancias no se define en esta etapa.

3.1. Estructura y Componentes de los URIs Base

El sistema solo define la autoridad para el Modelo, ya que las instancias de datos se generarán y se alojarán posteriormente. Se utiliza el formato Hash URI para el Modelo (`#`).

URIs para Clases y Propiedades (Modelo/Ontología)

La estructura se basa en el prefijo ns de la ontología (Ej: http://smartcity.es/alquiler#Barrio).

Dominio http://smartcity.es/
+
Ruta del Modelo alquiler
+
Hash URI #
+
Clase o Propiedad [ns:Nombre]

Los prefijos del modelo utilizados son:

Elemento URI Base (Namespace) Prefijo Uso
Ontología (Modelo) http://smartcity.es/alquiler# ns: Identifica Clases y Propiedades (ej: ns:Barrio) tal como se define en ontology.ttl.
Vocabulario de Modularidad https://w3id.org/mod# mod: Usado para metadatos de creación del modelo.
Marco de descripción de recursos http://www.w3.org/1999/02/22-rdf-syntax-ns rdf: Usado para describir nuestra Ontología
Definición de esquema XML http://www.w3.org/2001/XMLSchema xsd: Usado para definir el tipo de cada elemento (xsd:string)

3.2. Patrones de Nombrado (Modelo Ontológico)

Se eligen patrones significativos y persistentes que identifican unívocamente a cada elemento dentro del modelo.

3.2.1. Clases y Propiedades (Ontología - Prefijo ns)

  • Clases (Ej: ns:Barrio, ns:Renta): Se utiliza mayúscula inicial, singular y separación de palabras con "_" para los nombres de clases.
  • Propiedades (Ej: ns:tieneRentaMedia, ns:año_de_estudio): Se utiliza minúsculas y separación de palabras con "_" para propiedades de Objeto y de Dato.

Justificación: El uso del Hash URI garantiza que la ontología completa sea accesible bajo un único recurso (http://smartcity.es/alquiler).

3.2.2. Creación de Instancias (Patrón a Futuro)

Aunque la ontología no contiene instancias, se define el patrón que se utilizará en la fase de transformación de datos para asegurar la unicidad y la persistencia de los recursos creados. Se utilizará un resource path diferenciado.

Entidad a Modelar Patrón de URI Propuesto Ejemplo de URI (a futuro)
Barrio http://data.smartcity.es/alquiler/barrio/[nombre_normalizado] http://data.smartcity.es/alquiler/barrio/alipark
Métrica Trimestral http://data.smartcity.es/alquiler/metrica/[barrio]/[año]/[trimestre] http://data.smartcity.es/alquiler/metrica/alipark/2020/4

3.3. Estrategia de Desreferenciación

La desreferenciación es lo que hace que nuestros URIs (como http://smartcity.es/alquiler#Barrio) no sean solo un nombre, sino una dirección real en Internet a la que puedes "preguntar". Tienes que poder consultarlos para saber qué significan.

Como el que pregunta puede ser tanto una persona como una máquina, usamos la Negociación de Contenido para darles a cada uno lo que quiere.

Solicitud InPut OutPut
HTML (text/html) Una persona (navegador) Una página web legible para el usuario (Documentación de la Ontología)
RDF/Turtle (text/turtle) Un programa (software Linked Data) Se devuelve el Grafo RDF con la definición de la ontología (el código .ttl)

Esta estrategia aplica al Modelo (ns:) para asegurar que cualquiera que use el vocabulario obtenga la definición que necesita para la interoperabilidad.
